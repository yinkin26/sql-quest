üéØ The automatic filler! DEFAULT constraint provides fallback values when no data is specified during insertion.

üìå What is a DEFAULT Constraint?

A DEFAULT constraint specifies a default value for a column when no value is provided during an INSERT operation. It's like having a ü§ñ helpful assistant that fills in missing information automatically!

üéØ DEFAULT Constraint Features

üîç Key Features

  • üéØ AUTOMATIC VALUES - Fills in missing data during INSERT
  • üîÑ FLEXIBLE TYPES - Supports literals, expressions, and functions
  • ‚ö° PERFORMANCE FRIENDLY - No overhead during SELECT operations
  • üõ°Ô∏è DATA CONSISTENCY - Ensures columns always have meaningful values
  • üìä BUSINESS LOGIC - Encodes common default behaviors

üß† Basic Syntax

Method 1: Column Definition with DEFAULT

CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL,
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    login_count INT DEFAULT 0
);

Method 2: Adding DEFAULT to Existing Column

-- Add DEFAULT to existing column
ALTER TABLE users
ALTER COLUMN status SET DEFAULT 'pending';

-- Remove DEFAULT from column
ALTER TABLE users
ALTER COLUMN status DROP DEFAULT;

üîç Types of DEFAULT Values

üìù Literal Values

üìÖ Date and Time Functions

-- Current date

-- Current time

üî¢ Mathematical Expressions

-- System functions

üîÑ INSERT Behavior with DEFAULT

-- Method 1: Explicit DEFAULT keyword
INSERT INTO users (username, email, status, role)
VALUES ('john_doe', 'john@email.com', DEFAULT, DEFAULT);

-- Method 2: Omit columns (uses DEFAULT automatically)
INSERT INTO users (username, email)
VALUES ('jane_doe', 'jane@email.com');

-- Method 3: Specify some values, omit others
INSERT INTO users (username, email, role)
VALUES ('admin_user', 'admin@email.com', 'admin');

-- Method 4: Override DEFAULT with explicit value
INSERT INTO users (username, email, status, is_active)
VALUES ('test_user', 'test@email.com', 'inactive', FALSE);

‚ö†Ô∏è DEFAULT Constraint Best Practices

‚úÖ Do's

  • üéØ Meaningful defaults - choose values that make business sense
  • üìä Common values - use the most frequent expected values
  • üîÑ Status fields - perfect for workflow states
  • üìÖ Timestamps - use CURRENT_TIMESTAMP for audit trails
  • üî¢ Counters - start numeric counters at 0
  • ‚úÖ Boolean flags - set logical default states

‚ùå Don'ts

  • üö´ Don't use complex expressions - keep defaults simple
  • ‚ö° Don't use expensive functions - avoid performance impact
  • üîÑ Don't use non-deterministic functions - unless specifically needed
  • üéØ Don't set defaults for PRIMARY KEY - use AUTO_INCREMENT instead
  • üìä Don't use defaults for required business data - force explicit values

üí° Helpful Tips

  • üîç Use SHOW CREATE TABLE table_name; to see DEFAULT values
  • ‚ö° Performance - DEFAULT values don't impact SELECT performance
  • üõ°Ô∏è Data consistency - ensures columns always have values
  • üéØ Business logic - encode common behaviors in schema
  • üîÑ Maintenance - easier to change defaults than update existing data
  • üìä Documentation - defaults serve as implicit documentation
  • üïê Timestamps - CURRENT_TIMESTAMP is perfect for audit trails

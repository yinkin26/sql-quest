🛡️ SQL Constraints Overview

🛡️ Master the guardians of data integrity! Constraints are your database's security system.

📌 What are SQL Constraints?

SQL constraints are rules applied to table columns to enforce data integrity and ensure the accuracy and reliability of data in the database. Think of them as 🚦 traffic rules for your data - they prevent invalid data from entering your tables and maintain consistency across your database.

🎯 Why Use Constraints?

  • 🛡️ Data Integrity - Ensure data follows business rules
  • 🚫 Prevent Invalid Data - Stop bad data before it enters the database
  • 🔗 Maintain Relationships - Keep related tables synchronized
  • ⚡ Improve Performance - Some constraints create indexes automatically
  • 📋 Document Business Rules - Constraints serve as living documentation

🗂️ Types of SQL Constraints

🔑 PRIMARY KEY

Uniquely identifies each row in a table. Only one per table.

id INT PRIMARY KEY

🔗 FOREIGN KEY

Links two tables together, ensuring referential integrity.

FOREIGN KEY (dept_id) REFERENCES departments(id)

⭐ UNIQUE

Ensures all values in a column are different.

email VARCHAR(100) UNIQUE

🚫 NOT NULL

Prevents empty values in a column.

name VARCHAR(50) NOT NULL

✅ CHECK

Validates data against a specific condition.

CHECK (age >= 18)

🎯 DEFAULT

Provides a default value when none is specified.

status VARCHAR(20) DEFAULT 'active'

🔧 Adding Constraints to Existing Tables

You can add constraints to existing tables using ALTER TABLE:

-- Add a primary key
ALTER TABLE table_name
ADD CONSTRAINT pk_id PRIMARY KEY (id);

-- Add a foreign key
ALTER TABLE orders
ADD CONSTRAINT fk_customer
FOREIGN KEY (customer_id) REFERENCES customers(id);

-- Add a check constraint
ALTER TABLE products
ADD CONSTRAINT chk_price
CHECK (price > 0);

📋 Constraint Examples Showcase

🌍 Real-World Constraint Scenarios

🛒 E-commerce Database

-- Products table with business rules
CREATE TABLE products (
    id INT PRIMARY KEY AUTO_INCREMENT,
    sku VARCHAR(50) UNIQUE NOT NULL,           -- 📦 Unique product code
    name VARCHAR(200) NOT NULL,                -- 📝 Required product name
    price DECIMAL(10,2) CHECK (price > 0),     -- 💰 Price must be positive
    stock_quantity INT DEFAULT 0,              -- 📊 Default stock level
    category_id INT,                           -- 🗂️ Link to categories
    is_active BOOLEAN DEFAULT TRUE,            -- 🎯 Product status
    FOREIGN KEY (category_id) REFERENCES categories(id)
);

👥 User Management System

-- Users table with validation rules
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,      -- ⭐ Unique username
    email VARCHAR(100) UNIQUE NOT NULL,        -- 📧 Unique email
    password_hash VARCHAR(255) NOT NULL,       -- 🔒 Required password
    age INT CHECK (age >= 13 AND age <= 120),  -- ✅ Age validation
    role VARCHAR(20) DEFAULT 'user',           -- 👤 Default role
    is_verified BOOLEAN DEFAULT FALSE,         -- ✅ Email verification
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

⚠️ Constraint Considerations

🚨 Important Notes

  • ⚡ Performance Impact - Constraints add validation overhead
  • 🔒 Data Migration - Existing data must satisfy new constraints
  • 🚫 Constraint Violations - Invalid data insertions will be rejected
  • 🔗 Cascading Effects - Foreign key constraints can prevent deletions
  • 📝 Naming Conventions - Use descriptive constraint names

💡 Best Practices

  • 🎯 Plan constraints early - Design them with your table structure
  • 📝 Use descriptive names - pk_users_id, fk_orders_customer_id
  • 🧪 Test with sample data - Ensure constraints work as expected
  • 📚 Document business rules - Constraints should reflect real requirements
  • ⚡ Consider performance - Too many constraints can slow operations
  • 🔄 Review regularly - Business rules change over time

🧠 Summary

  • 🛡️ Constraints enforce data integrity and business rules
  • 🔑 Primary keys uniquely identify rows
  • 🔗 Foreign keys maintain relationships between tables
  • ⭐ Unique constraints prevent duplicate values
  • 🚫 NOT NULL ensures required data is provided
  • ✅ CHECK constraints validate data against conditions
  • 🎯 DEFAULT values provide fallback data
